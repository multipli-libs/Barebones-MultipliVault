## Auditor Notes & Key Considerations

Pointers that we feel are important to auditors. We recommend reading the `README.md` before reading this section.

- Implementation Details
    - `lastPricePerShare` is always denominated in 1e18, regardless of token decimals
    - All `preview*` methods return values after fee calculations

- Architecture Questions
    - **Mixed Access Control**: Most `VaultFundManager` methods require `MultipliVault.manage()` calls, but `flashRedeem()` can be called directly by operators
    - **Concurrency Risk**: Fund managers and operators may call methods simultaneously. Will this lead to any issues? 
    - **Design Question**: Should `flashRedeem` be in a separate contract?
    - **Complexity**: Is the current access control for `VaultFundManager` methods overly complex?
    - **Workflows**: Go through "workflow", "Backend Monitoring and Operations" and "VaultFundManager: Operational Management Layer" sections to understand the workflow. Are there any possible issues that might arise from operating the vault in the specified manner. 

- Security Concerns

    - **FlashRedeem**
        ⚠️ **We suspect flashRedeem may contain exploits. Please allocate extra time for analysis.**

    - **Reentrancy Protection**
        - Reentrancy guards added to `mint()`, `deposit()`, and `flashRedeem()`
        - **Risk**: Operators could reenter vault during `flashRedeem` to acquire shares at manipulated prices
        - **Questions**: Are guards sufficient? Can exploits be prevented without them?

    - **3.3 UUPS Implementation**
        - **Verify**: UUPS pattern correctly implemented in `MultipliVault`
        - **Review**: Deployment script for proper UUPS setup and upgrade mechanisms
        - **Check**: Future upgrade compatibility and storage layout safety

- Code Cleanup
    - This block in `MultipliVault.flashRedeem()` may be redundant:
    ```solidity
    // totalSupply check was added before reentrancy guard - still needed?
    require(totalSupply() == totalSupplyBefore - shares, Errors.TotalSupplyMismatch());
    ```
    - In flashRedeem, a lot of state checks have been currently coded. Are all of them required? Gas will not be limiting factor here since Operators are expected to hold and unwind positions in millions so extra $20-$30 might not be an issue.

- Code Organization and Dependency management
    - Please suggest if you feel there are improvements that can be made to the way files are organised or the way dependencies are managed. 
- Gas optimisations
    - Please suggest if there any areas for gas optimisations in the most commonly used methods (eg: deposit, mint, preview methods, and others)
